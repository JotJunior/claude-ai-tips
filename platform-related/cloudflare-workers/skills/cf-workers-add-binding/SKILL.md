# Cloudflare Workers: Adicionar Binding

Adiciona binding de recurso ao `wrangler.toml` e sincroniza a interface `Env`
TypeScript. O binding torna o recurso acessivel via `env.NOME` no codigo do
Worker, sem necessidade de variaveis de ambiente manual.

## Tabela de Bindings Suportados

| Tipo | Secao wrangler.toml | Acesso em codigo | Uso comum |
|------|---------------------|-------------------|-----------|
| D1 Database | `[[d1_databases]]` | `env.DB` | SQLite query |
| KV Namespace | `[[kv_namespaces]]` | `env.KV` | key-value cache |
| R2 Bucket | `[[r2_buckets]]` | `env.BUCKET` | object storage |
| Queue Producer | `[[queues.producers]]` | `env.QUEUE` | send messages |
| Queue Consumer | `[[queues.consumers]]` | `env.QUEUE` | receive messages |
| Durable Object | `[[durable_objects.bindings]]` | `env.MYDO` | single-instance state |
| AI | `[ai]` | `env.AI` | inference Gemma/llama |
| Vectorize | `[[vectorize]]` | `env.VECTORIZE` | vector search |

## Pre-Flight Reads

1. **`wrangler.toml`** atual — identificar secao do binding existente (se houver)
2. **`src/index.ts`** ou **`worker/**/*.ts`** — localizar interface `Env` existente
3. **`worker-configuration.d.ts`** se ja existir — ver estado dos tipos gerados
4. Confirmar que o recurso subjacente ja foi criado:
   - D1: `wrangler d1 list` deve mostrar o database
   - KV: `wrangler kv namespace list` deve mostrar o namespace
   - R2: `wrangler r2 bucket list` deve mostrar o bucket

## Workflow: Adicionar Binding

### Passo 1 — Identificar o tipo e obter metadata

Cada tipo de binding tem workflow diferente para obter o ID/nome.

**D1:**
```bash
wrangler d1 list
# Output: name=<nome>  id=<uuid>  version=...
```

**KV:**
```bash
wrangler kv namespace list
# Output: name=<nome>  id=<uuid>
```

**R2:**
```bash
wrangler r2 bucket list
# Output: name=<nome>
# R2 nao tem id global — usa o nome diretamente
```

**Queue:**
```bash
wrangler queues list
# Output: queue name, consumer count
```

**Durable Object:**
```bash
# DO precisa de um script ja deployado + class name
wrangler deployments list 2>/dev/null | grep "Durable Object"
```

### Passo 2 — Editar wrangler.toml

```toml
name = "my-worker"
main = "src/index.ts"
compatibility_date = "2024-01-01"

# D1 binding
[[d1_databases]]
  binding = "DB"                    # nome usado no env — UPPERCASE convensao
  database_name = "my-database"   # nome returned by wrangler d1 create
  database_id = "abc-123-def"     # UUID returned by wrangler d1 create
  # preview_id = "xyz-789"         # para wrangler dev local (opcional se id=d1 local)

# KV binding
[[kv_namespaces]]
  binding = "CACHE"
  id = "abc-123-def"               # para production
  # preview_id = "xyz-789"        # para wrangler dev (cria namespace separado)

# R2 binding
[[r2_buckets]]
  binding = "ASSETS"
  bucket_name = "my-assets"       # nome do bucket

# Queue producer
[[queues.producers]]
  binding = "EVENTS"
  queue = "events-queue"           # nome da queue

# Queue consumer
[[queues.consumers]]
  queue = "events-queue"
  script = "my-worker"            # opcional — sobrescreve worker name
  max_batch_size = 10             # msgs por batch
  max_batch_timeout = 30          # segundos

# Durable Object
[[durable_objects.bindings]]
  name = "MY_DO"                  # nome no env — UPPERCASE
  class_name = "Counter"          # nome da classe DO no codigo
  script = "my-worker"            # opcional

# AI
[ai]
  binding = "AI"

# Vectorize
[[vectorize]]
  binding = "VECTORIZE"
  index_name = "embeddings-index"
```

### Passo 3 — Gerar tipos TypeScript

```bash
wrangler types
```

Isso gera/atualiza `worker-configuration.d.ts`:

```typescript
// Generated by Wrangler
// run `wrangler types` to update

interface Env {
  DB: D1Database;
  CACHE: KVNamespace;
  ASSETS: R2Bucket;
  EVENTS: Queue;
  AI: Ai;
  VECTORIZE: VectorizeIndex;
}
```

### Passo 4 — Atualizar interface Env no codigo

Se voce tem interface `Env` custom, **nao sobrescreva** — faca merge:

```typescript
// src/index.ts ou worker/types.ts
interface Env {
  // bindings gerados (mantidos por wrangler types)
  DB: D1Database;
  CACHE: KVNamespace;
  ASSETS: R2Bucket;
  EVENTS: Queue;

  // Suas vars custom (SECRET, etc.)
  API_SECRET: string;

  // DO binding custom
  MY_DO: DurableObjectNamespace<Counter>;
}
```

### Passo 5 — Usar no codigo

```typescript
export default {
  async fetch(request: Request, env: Env): Promise<Response> {
    // D1
    const result = await env.DB
      .prepare("SELECT * FROM users WHERE id = ?")
      .bind(userId)
      .first();

    // KV
    await env.CACHE.put("key", JSON.stringify(data), {
      expirationTtl: 3600,
    });
    const cached = await env.CACHE.get("key");

    // R2
    await env.ASSETS.put("file.txt", body);
    const object = await env.ASSETS.get("file.txt");

    return new Response("OK");
  },
};
```

## Bindings por Ambiente ([env])

Para production vs preview, separe via secoes `[env.production]`:

```toml
[env.production]
[[d1_databases]]
  binding = "DB"
  database_name = "prod-db"
  database_id = "prod-uuid"

[[kv_namespaces]]
  binding = "CACHE"
  id = "prod-kv-id"

[env.preview]
[[d1_databases]]
  binding = "DB"
  database_name = "preview-db"
  database_id = "preview-uuid"

[[kv_namespaces]]
  binding = "CACHE"
  preview_id = "preview-kv-id"
```

## Exemplo BOM

```toml
# wrangler.toml
[[d1_databases]]
  binding = "DB"
  database_name = "auth-db"
  database_id = "f2a8c4d1-5e3b-4a9c-8d7f-1234567890ab"
  preview_id = "local-preview-id"    # nao usar id de producao em dev
```

```typescript
// worker/types.ts — interface completa
export interface Env {
  DB: D1Database;
  CACHE: KVNamespace;
  ASSETS: R2Bucket;
  EVENTS: Queue;
  API_KEY: string;          // via `wrangler secret put API_KEY`
}
```

## Exemplo RUIM

```toml
# WRONG: binding name em minusculo
[[d1_databases]]
  binding = "db"   # Nao seguir convencao — deve ser UPPERCASE
```

```toml
# WRONG: usando id de production em preview (expoe creds)
[env.preview]
[[d1_databases]]
  binding = "DB"
  database_id = "prod-uuid"   # WRONG — preview_id nao especificado
```

```typescript
// WRONG: redeclarar Env sem merge (wrangler types sobrescreve)
interface Env {   // PERDE os tipos gerados automaticamente
  API_KEY: string;
}
```

## Gotchas

### 1. Binding names UPPERCASE — convencao Cloudflare

Wrangler e SDKs esperam `env.DB`, `env.CACHE`, `env.ASSETS`. Bindings em
minusculo podem funcionar no runtime mas quebram `wrangler types` e geram
warnings em deploy.

### 2. preview_id vs id — sepacao prod/dev

`id` eh usado em `wrangler deploy`. `preview_id` (ou `kv_namespaces.preview_id`,
`d1_databases.preview_id`) eh usado em `wrangler dev`. Se nao especificar
`preview_id`, `wrangler dev` usa `id` como fallback (nao recomendado —
aponta pra producao).

### 3. Rodar `wrangler types` apos cada mudanca

Qualquer adicao/remocao de binding em `wrangler.toml` requer novo `wrangler
types` para atualizar `worker-configuration.d.ts`. Esquecer de rodar resulta
em `Binding not found` em runtime.

### 4. DO migration tag (Durable Objects)

Quando renomear uma classe DO, use `migrations` no bloco DO para atualizar
instancias existentes:

```toml
[[durable_objects.migrations]]
  tag = "v1"           # label da migracao
  new_classes = ["Counter"]  # novas classes
```

A cada rename/reclass, adicionar novo tag. O DO antigo permanece acessivel
ate que todas as instancias migrem.

### 5. Wrangler types nao faz merge automatico

`wrangler types` sobrescreve `worker-configuration.d.ts` inteiro. Se voce
edita o arquivo manualmente, suas mudancas serao perdidas. Usar abordagem:
**types gerados como source of truth + interface manual via merge**.

### 6. AI binding — um por Worker

Cada Worker pode ter **apenas um** binding `[ai]`. Se precisar de multiplos
models, use o mesmo binding `env.AI` com diferentes `run()` calls specifying
model.

### 7. Queue binding — producer e consumer no mesmo Worker

Um Worker pode ter ambos `[[queues.producers]]` e `[[queues.consumers]]`
para a mesma queue. Nesse caso, o mesmo script envia e recebe mensagens.
Cuidado com loops infinitos (consumer re-envia para mesma queue).

## Quando NAO Usar

- **Criar o recurso D1/KV/R2/Queue** — use a skill especifica: `cf-workers-create-d1`,
  `cf-workers-create-kv-namespace`, `cf-workers-create-r2-bucket`
- **Configurar secrets (API keys, tokens)** — use `wrangler secret put` diretamente;
  bindings nao sao para secrets (use env padrao)
- **Escrever queries SQL** — isso pertence a `data-related/d1/`
- **Configurar FTS5 (Full-Text Search)** em D1 — `data-related/d1/` skill
- **AI fine-tuning ou treinamento** — skill especifica de AI
- **Provisionar Durable Object class** — Durable Objects precisam de script
  separado deployado primeiro; bindingsso habilitam o acesso
